# 多 Agent 矩阵搭建实战 - 从 0 到 1 构建 AI 开发团队

## 选题背景

很多人用多 Agent 只是"开了多个对话框"：
- 没有角色定义和边界
- 没有知识库沉淀
- 没有工作流自动化
- 每次会话都是重新开始

需要一篇从 0 到 1 的实战指南，教读者搭建真正的 AI 开发团队。

## 核心架构

### 5 个核心角色

| 角色 | 职责 | 不可为 |
|------|------|--------|
| 项目负责人 | 任务路由 | 不阅读代码 |
| 团队主管 | 模块拆解与协调 | 不进行实现 |
| 领域专家 | 组件/样式/测试等 | 不越界处理其他领域 |
| 图书管理员 | 记录跨会话决策 | 不自动应用变更 |
| 橡皮鸭 | 引导思考而非给答案 | 不直接编写代码 |

### 知识库结构

```
knowledge/
├── quick-reference.md (速查表)
├── patterns/ (各領域模式文件)
├── project/ (项目上下文)
├── templates/ (PR 模板等)
└── accs/ (验收标准)
```

## 核心观点

> AI 输出内容的质量，与你为其提供的上下文质量成正比。

> 结构化胜过非结构化智能。一个组织良好的专家团队，即使底层是同一模型，其产出结果也始终优于单一通用助手。

> 约束使专业化成为现实。

> 记忆具有复利效应。

> 引导胜于生成。

## 实施步骤

1. 记录反模式 → 创建速查表
2. 添加 1-2 个专家角色 → 编写 SKILL.md
3. 创建记忆文件 → 记录跨会话决策
4. 自动化一个工作流 → 如/preflight
5. 逐步扩展 → 感受缺口后再添加

## 跨平台适配

- **Claude Code**：原生支持子代理、技能、知识库（最完整）
- **Copilot**：读取 .claude/skills/目录，支持自定义代理
- **Codex**：支持技能文件，但不支持子代理路由

> 知识库是最具可移植性的层面——模式文件仅是 Markdown，随处可用。
