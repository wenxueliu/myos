# 多 Agent 协作的成本与性能优化 - RAM 瓶颈与 Agent 选择

## 选题背景

很多人以为多 Agent 的瓶颈是：
- token 成本
- API 速率限制

但真实案例显示：**瓶颈是 RAM**。

需要一篇性能优化指南，帮助读者合理配置硬件和选择 Agent。

## 真实成本数据

| 项目 | 费用 |
|------|------|
| Claude API | $100/月 |
| Codex API | $90/月 |
| **总计** | **$190/月** |
| 新手起步 | $20/月 |

> 不是 token 成本，不是 API 速率，而是内存。

## RAM 瓶颈分析

**每个 Agent 需要**：
- 自己的 worktree
- 自己的 node_modules
- 运行构建、类型检查、测试

**5 个 Agent 同时跑** = 5 个并行的 TypeScript 编译器 + 5 个测试运行器 + 5 套依赖加载到内存。

**实测数据**：
- Mac Mini (16GB RAM)：最多同时跑 4-5 个 Agent
- Mac Studio M4 Max (128GB RAM)：可跑更多（$3500）

## Agent 选择策略

| Agent | 擅长领域 | 成本 | 使用占比 |
|-------|----------|------|----------|
| Codex (gpt-5.3-codex) | 后端逻辑、复杂 bug、多文件重构 | $90/月 | 90% |
| Claude Code (claude-opus-4.5) | 前端工作、权限问题少 | $100/月 | 速度型 |
| Gemini | 设计师，有设计审美 | 免费 | UI 设计 |

> Gemini 设计，Claude 建造。

## 性能优化技巧

### 1. 并发控制
- 限制同时运行的 Agent 数量（根据 RAM）
- 使用队列管理任务

### 2. 监控设计
- cron 每 10 分钟检查（100% 确定性，省 token）
- 只在需要人工介入时通知

### 3. 失败重试
- 最多重试 3 次
- 每次重试动态调整 prompt（不是重复执行）

### 4. 资源清理
- 每天 cron 清理孤立的 worktree 和任务记录
- 避免资源泄漏

## 核心观点

> 这个监控脚本是 100% 确定性的，非常省 token，只在需要人工介入时才会通知作者。

> 成功的模式会被记录下来。时间越长，编排层写的 prompt 越好，因为她记得什么能成功。

## 硬件建议

| 规模 | 推荐配置 | 成本 |
|------|----------|------|
| 新手起步 | 任意云主机 (4GB RAM) | $20/月 |
| 中度使用 | Mac Mini/云主机 (16GB RAM) | $190/月 |
| 重度使用 | Mac Studio M4 Max (128GB RAM) | $3500 + $190/月 |

## 实施步骤

1. 评估当前 RAM 容量
2. 设置最大并发 Agent 数
3. 配置任务队列
4. 编写监控脚本（省 token 设计）
5. 设置资源清理任务
