# CLAUDE.md 编写最佳实践 - 200 行限制下的高效指令设计

## 选题背景

CLAUDE.md 会占用上下文窗口，编写方式直接影响 AI 的遵循效果：
- 每个文件最好不要超过 200 行
- 模糊指令容易被忽略
- 冲突指令会导致执行混乱

需要一篇 CLAUDE.md 编写最佳实践指南。

## 核心原则

### 1. 控制篇幅

> 每个 CLAUDE.md 文件最好不要超过 200 行。文件过长会消耗大量 token，可能导致 Claude 难以完全遵循指令。

### 2. 结构清晰

```markdown
# 项目约定

## 构建命令
- 开发环境：`pnpm dev`
- 构建：`pnpm build`
- 测试：`pnpm test`

## 代码规范
- 使用 2 空格缩进
- 组件放在 `src/components/`
```

### 3. 具体明确

| ❌ 模糊指令 | ✅ 明确指令 |
|------------|------------|
| "格式化代码" | "使用 2 空格缩进" |
| "测试你的改动" | "提交前运行 pnpm test" |
| "保持文件组织" | "API 处理函数放在 src/api/handlers/" |

### 4. 避免冲突

多个 CLAUDE.md 文件中存在相互矛盾的指令，Claude 可能会选择其中之一执行。

## 高级技巧

### @语法导入

```markdown
# 项目概述
参见 @README 获取项目简介

# 可用命令
参见 @package.json 了解所有 npm scripts
```

- 最多支持 5 层嵌套
- 相对路径相对于包含 `@` 的文件解析

### 路径特定规则

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 开发规则
- 所有 API 端点必须包含输入校验
```

### 模块化组织

```
.claude/rules/
├── code-style.md
├── testing.md
├── api-design.md
└── security.md
```

## 核心观点

> 指令要尽量具体，最好可以直接验证执行结果。模糊的说明容易被忽略或理解错误。

> 使用 markdown 的标题和列表来组织内容。Claude 会像人一样读取结构化信息。

1. **为什么需要控制篇幅** - 上下文窗口的限制
2. **4 个编写原则** - 篇幅、结构、明确、一致
