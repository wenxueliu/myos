# OpenClaw 编排层设计指南 - 业务上下文与 Agent 调度

## 选题背景

很多人用 OpenClaw 只是"多了一个对话框"：
- 没有业务上下文沉淀
- 没有 Agent 调度策略
- 没有监控和重试机制

需要一篇编排层设计的实战指南。

## 编排层核心职责

| 职责 | 说明 | 示例 |
|------|------|------|
| **持有业务上下文** | 会议记录、客户数据、历史决策 | Obsidian 自动同步 |
| **翻译 prompt** | 把业务需求翻译成精确的技术指令 | 客户原话→技术规格 |
| **调度 Agent** | 根据任务类型选择合适的执行者 | 后端→Codex, 前端→Claude |
| **监控与重试** | 检查进度、分析失败、动态调整 prompt | cron 每 10 分钟检查 |

## 安全边界设计

```
编排层 (OpenClaw)
├── 读取 Obsidian 会议记录
├── 访问生产数据库 (只读)
├── 管理员 API 权限
└── 调度 Agent

执行层 (Agent)
├── 读写代码库
├── 运行测试
├── 提交代码
└── 创建 PR
❌ 不接触生产数据库
❌ 不看客户敏感信息
```

## 动态 Prompt 调整

**坏例子（静态 prompt）**：
```
"实现自定义模板功能"
```

**好例子（动态调整）**：
```
"停。客户要的是 X，不是 Y。这是他们在会议里的原话：
'我们希望保存现有配置，而不是从头创建新的。'
重点做配置复用，不要做新建流程。"
```

> 当 Agent 失败时，编排层不会用同样的 prompt 重启。她会带着完整的业务上下文，分析失败原因，然后重写 prompt。

## Agent 选择策略

| Agent | 擅长领域 | 使用场景 |
|-------|----------|----------|
| Codex (gpt-5.3-codex) | 后端逻辑、复杂 bug、多文件重构 | 90% 的任务 |
| Claude Code (claude-opus-4.5) | 前端工作、权限问题少 | 速度型选手 |
| Gemini | 设计师，有设计审美 | UI 设计 |

## 监控设计

**检查项**（每 10 分钟）：
- tmux 会话还活着吗？
- 有没有创建 PR?
- CI 状态如何？
- 如果失败了，是否需要重试？（最多 3 次）

> 这个监控脚本是 100% 确定性的，非常省 token，只在需要人工介入时才会通知作者。

## 核心观点

> 通过上下文的专业化分工，而不是换更强的模型。

> 时间越长，编排层写的 prompt 越好，因为她记得什么能成功。

## 实施步骤

1. 设置 Obsidian 自动同步（会议记录→业务上下文）
2. 编写 Agent 选择逻辑（任务类型→Agent 映射）
3. 创建监控脚本（cron + tmux 状态检查）
4. 设计失败重试机制（分析原因→动态调整 prompt）
5. 建立成功模式记录（什么 prompt 结构有效）
