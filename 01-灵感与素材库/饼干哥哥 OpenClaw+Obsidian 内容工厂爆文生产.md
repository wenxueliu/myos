# 用 OpenClaw+Obsidian 搭建内容工厂，写 100W+ 阅读爆文，单篇 6000 元

> 去年，我做过公众号爆文项目，凭借以前做内容的网感，爆了一篇百万级阅读量的文章，单篇直接带来了 6000 多元的收益。

---

## 🎯 验证逻辑：从流量驱逐到数据壁垒

为了验证爆文逻辑，我单独开了一个全新的号「第二曲线增长」，用纯粹的数据和工业化逻辑重新做一遍：

1. 抓了 28 篇细分赛道的高赞爆款内容
2. 让模型严格划分内容类型、情绪爆点和文章结构
3. 拉出 100 个痛点选题
4. 按照固定写作风格和提示词框架生成文章
5. 连标点符号都不改，直接复制粘贴发布
6. 封面图都是同一张

**结果**：发到第四篇时，文章直接触发推荐算法，开始疯狂进池子爆流量。

> 但这种做法太驱逐流量了，没有任何复利，也没有任何个人资产沉淀。

**现在**：把这套逻辑平移到智能体架构上，用 OpenClaw 配合 Obsidian，效果不仅好十倍，而且能真正建立长期的数据壁垒。

## 🤔 01 为什么是 OpenClaw + Obsidian？

### 碎片化使用的问题

很多人每天都在用各种大模型，但效率没有质变：
- 想到一个点子，去问 AI
- 得到一篇文章，复制发布，然后关闭网页
- 下次再有想法，又得重新设定背景、重新调教语气
- 每次都在重复无效劳动

### 系统化创作应该是这样的

1. 你有一个想法，记录进本地的选题库
2. AI 接收到指令，自动检索你积累了几个月的爆款素材库
3. 找到匹配的理论框架后，复用已被市场验证过的内容结构进行写作
4. 文章发布后，数据表现再次沉淀到方法论里

### 两个核心组件

| 组件 | 作用 | 说明 |
|------|------|------|
| **Obsidian** | 持久化知识数据库 | 完全本地化的 Markdown 知识管理软件，双向链接把孤立文本连接成立体知识网络 |
| **OpenClaw** | 后台自动执行引擎 | 7x24 小时静默运行的守护进程，通过手机接口发指令，自动读取 Obsidian 文件执行检索、拼接和写作 |

> 这个组合，无论是个人 IP 博主、公众号创作者，还是跨境电商企业，都是一套极具统治力的基础设施。

## 📊 02 内容生产流程被彻底重构

**场景示例**：

1. 走在路上看到优质行业报道，随手复制链接发给手机上的 Agent
2. OpenClaw 云端静默抓取正文，提取核心逻辑，打上 YAML 标签，同步到本地 Obsidian 灵感库
3. 晚上发一句指令开始处理灵感，Agent 自动调取素材给选题角度（可实时搜索）
4. 生成大纲、写文章，期间任何问题都可以更新记忆
5. 后续想改，直接改 md 文件

## ⚙️ 03 配置本地底层系统

### 安装 Obsidian

1. 官网下载 Obsidian
2. 本地新建空文件夹作为仓库（Vault），如 `bgggcontent`

### ⚠️ 关键安全配置

**绝对不能让 OpenClaw 直接操作这个文件夹！**

原因：
- 大模型会调用原生 `mv` 指令移动文件
- Obsidian 依赖双向链接，文件被强行移动会导致所有内部链接失效
- 整个知识系统会出现大量死链

**解决方案**：

```bash
# 安装 obsidian-cli 命令行工具
npm install -g obsidian-cli
```

并在 OpenClaw 中严格挂载官方的 `obsidian-skills`

> 这个工具确保 AI 在移动/删除笔记时，会向 Obsidian 底层引擎发送信号，全局引用关系自动更新。

## 🔗 04 架构设计：软链接打通多 Agent 矩阵

### 问题

内容中台的终极形态是多个专职 Agent：
- 研究员：全网监控抓取素材
- 小红书专员：撰写种草文案
- 审稿总管：统筹全局

但 OpenClaw 安全沙盒机制严格，每个 Agent 只能访问自己工作区内的文件。

### 最佳实践：软链接（Symlink）

**1. 建立中立的物理存储**

```bash
/Users/binggan/Documents/bgggcontent
```

**2. 为各 Agent 建立专属通道**

```bash
ln -s /Users/binggan/Documents/bgggcontent /Users/binggan/.openclaw/workspace-lead/bgggcontent
ln -s /Users/binggan/Documents/bgggcontent /Users/binggan/.openclaw/workspace-xhs/bgggcontent
ln -s /Users/binggan/Documents/bgggcontent /Users/binggan/.openclaw/workspace-gzh/bgggcontent
```

**效果**：
- 素材库是唯一真实数据源
- 研究员把竞品分析丢进爆款素材库，公众号 Agent 零延迟读取并写作

## 📁 05 核心目录映射与内容流转规则

### 目录结构（遵循内容生产生命周期）

| 目录 | 用途 |
|------|------|
| `01-灵感与素材库` | 日常灵感剪报 + 爆款素材片段 |
| `02-选题池` | 人工确认或 AI 衍生的待写命题 |
| `03-内容工厂` | 大纲挑选区、初稿打磨区、终稿确认区 |
| `04-已发布归档` | 成品最终沉淀 |

### 强制元数据管理

**系统内严禁出现任何纯文本的裸文件！**

每篇新文档顶部必须携带标准 YAML 属性表头：

```yaml
---
标题：xxx
阶段：灵感/选题/大纲/初稿/终稿/已发布
关联参考：[[xxx]]
发布平台：公众号
---
```

> 这是 AI 识别文件状态的唯一索引标识。缺乏这块属性区，大模型会在图谱中彻底迷失方向。

## 📜 06 配置 Agent 执行准则

在公众号 Agent 工作区（`workspace-gzh`）建立四个核心规则文件：

### 1. SOUL.md - 安全防线底线声明

```markdown
## SOUL.md - 系统底层运行法则

1. 绝对的操作护栏：
   - 严禁使用 Linux 原生的 mv 或 rm 命令操作 bgggcontent 目录下的任何文件
   - 所有涉及文件的移动、重命名、删除，必须且只能调用 obsidian-cli
   - 必须严格遵循 obsidian-skills 规范生成 Markdown 文本

2. 强制元数据注入：
   - 在 bgggcontent 目录下创建任何新的.md 文件，文件最顶部必须包含标准 YAML 属性
   - 禁止生成纯文本的无属性文件

3. 目录越权禁止：
   - 活动范围仅限于当前工作区和软链接的 bgggcontent
   - 严禁尝试访问系统的其他目录或修改点前缀（如.git, .obsidian）的隐藏文件夹
```

### 2. USER.md - 个人风格设定

```markdown
## USER.md - 创作者偏好与风格设定

1. 平台调性：微信公众号。文章需要有深度，适合碎片化阅读，但逻辑要求严密。

2. 语言风格：
   - 拒绝常规 AI 词汇（如：总而言之、不可否认、在这个瞬息万变的时代）
   - 喜欢用短句，多用换行留白
   - 语气要求：真诚、专业、带一点极客的分享欲

3. 排版习惯：
   - 喜欢用>引用框来突出金句
   - 列表项一定要清晰，多用加粗**强调核心关键词
```

### 3. AGENTS.md - 每次启动必读列表

```markdown
## Every Session

- Before doing anything else:
  - Read SOUL.md — this is who you are
  - Read USER.md — this is who you're helping
  - Read memory/YYYY-MM-DD.md (today + yesterday) for recent context
  - If in MAIN SESSION: Also read MEMORY.md
  - Read SOP_GZH.md — this is your core content production workflow
```

### 4. SOP_GZH.md - 标准作业程序

```markdown
## SOP_GZH.md - 内容生产标准作业程序

当用户下达模糊指令时，对照以下阶段执行标准动作：

### 阶段 1：灵感捕获
- 触发：用户发送微信/小红书内容链接或原文
- 动作：整理保存至「01-灵感与素材库/1-日常灵感剪报/」，提炼 1-3 个选题存入「02-选题池/待写选题库/」

### 阶段 2：素材沉淀
- 触发：用户发送金句或特定逻辑框架
- 动作：打标签，存入「01-灵感与素材库/2-爆款素材片段/」

### 阶段 3：选题立项与大纲生成
- 触发：用户问今天写什么或要求开始写某选题
- 动作：列出待写选题库供选择 → 检索灵感库 → 生成 3 个不同切入点大纲 → 存入「03-内容工厂/1-大纲挑选区/」

### 阶段 4：初稿打磨
- 触发：用户确认某一大纲
- 动作：读取 USER.md 写作风格 → 检索爆款素材片段找金句润色 → 生成初稿存入「03-内容工厂/2-初稿打磨区/」

### 阶段 5-7：终稿确认与归档
- 触发：用户发回修改后的文本，说明是终稿并已发布
- 动作：使用 obsidian-cli move 移动至「04-已发布归档/公众号已发布/」，更新 YAML 状态为「已发布」
```
