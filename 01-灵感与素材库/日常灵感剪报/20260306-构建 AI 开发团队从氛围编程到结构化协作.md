# 构建一个 AI 开发团队：从"氛围编程"到结构化协作

**作者**：Tableau AI 智能践行  
**发布**：2026 年 3 月 1 日

---

## 核心问题：告别"氛围编程"

> 笔者曾尝试过所谓的"氛围编程 (Vibe-Coding)"。其诱惑力是真实的：直接粘贴 AI 生成的代码，运行它，修复出现的错误，再让 AI 修复，再次运行。结果来得快，但理解为零。

**氛围编程的终点**：
- AI 生成看似合理但毫无用处的工具函数
- 代码之所以"能运行"，仅仅是因为 AI 同时生成了实现和验证它的测试
- 你无法在代码评审中为这些代码辩护，因为你既没有编写它，也无法解释它的作用

> 笔者并非想停止使用 AI，而是希望停止低效地使用它。

---

## 核心理念：让 AI 像真正的开发团队一样工作

在一个真实的软件开发团队中，没有人包揽一切。团队里有负责规划、构建、评审和测试的人员。

**期望的模式**：不是一个试图包办一切的 AI，而是一组各司其职的专家：

| 角色 | 职责 |
|------|------|
| **项目负责人 (Project Owner)** | 将任务路由给合适的人员 |
| **团队主管 (Team Lead)** | 将功能拆解为模块并协调专家工作 |
| **领域专家 (Domain Specialists)** | 分别负责组件、样式、无障碍访问、类型和测试 |
| **图书管理员 (Librarian)** | 记录跨会话的决策 |
| **橡皮鸭 (Rubber Duck)** | 在我希望思考问题而非直接获取解决方案时提供帮助 |

## 成功关键：明确界定智能体的"不可为"

> 每个智能体定义中最有用的部分，并非其职责清单，而是其禁止事项清单。

| 角色 | 不可为 |
|------|--------|
| 项目负责人 | 不阅读代码，只负责路由 |
| 团队主管 | 不进行实现，只负责协调 |
| 质量保证工程师 | 不修复缺陷，只负责报告 |
| 图书管理员 | 不自动应用变更，只提出建议并等待确认 |
| 项目经理 | 不做技术决策，只负责格式化交付物 |

> 正是这些约束，使得专业化成为现实，而非仅仅是一个标签。

## "橡皮鸭"：对抗"氛围编程"的解药

> 当调用"橡皮鸭"时，我得到的不是代码，而是一系列问题。

**橡皮鸭的工作方式**：
- "这个组件在你的代码库中遵循什么模式？"
- "你检查过现有对话框如何处理焦点管理吗？"
- "无障碍访问模式文件中关于实时区域是怎么说的？"

> 它引导我查看相关文件和模式，将我推向正确的方向，但绝不替我编写代码。我始终掌握着方向盘。

**例外**：如果要求，它会编写单元测试——因为没有人需要被"引导"去编写第 47 个挂载包装器工厂函数。

## "图书管理员"：拥有记忆的系统

每次开始一个新的 AI 对话，你都会丢失一切。每一个偏好、每一个决策、每一次纠正，都消失了。

**图书管理员的机制**：
- 监听类似指令的语言（如"记住要"、"总是"、"从不"、"从现在开始"）
- 当检测到此类语言时，草拟一个结构化的指令并提交批准
- 未经明确确认，任何内容都不会写入指令文件

**指令文件示例**：
```yaml
DIR-001:
  规则："在使用 ARIA 属性之前，必须优先使用语义化 HTML"
  应用于：[无障碍访问专家，代码评审员]
DIR-002:
  规则："测试文件必须在 afterEach 中清理已挂载的组件"
  应用于：[测试专家，代码评审员]
DIR-003:
  规则："禁止硬编码颜色值 - 必须始终使用主题变量"
  应用于：[样式专家，代码评审员]
```

## 知识库体系

```
knowledge/
├── quick-reference.md          # "若见 X，则做 Y"速查表
├── patterns/
│   ├── component-patterns.md
│   ├── styling-patterns.md
│   ├── accessibility-patterns.md
│   ├── type-patterns.md
│   ├── testing-patterns.md
│   ├── i18n-patterns.md
│   ├── performance-patterns.md
│   └── debugging-patterns.md
├── project/
│   ├── project-context.md      # 架构与约定
│   └── developer-profile.md    # 个人工作偏好
├── templates/
│   └── pr-template.md
└── accs/                       # 各功能验收标准
    ├── feature-a-accs.md
    └── feature-b-accs.md
```

**速查表示例**：

| 若见... | 则做... | 严重等级 |
|----------|---------|----------|
| `any` 类型 | 替换为恰当类型或 `unknown` | 关键 |
| SCSS 中的 `@import` | 替换为 `@use` | 关键 |
| 带点击处理器的 `<div>` | 替换为 `<button>` | 高 |
| 无 role 的样式化列表 | 添加 `role="list"` | 高 |

## 自动化工作流

**/preflight** - PR 前三阶段流水线：
1. 代码检查（依据速查表进行模式检查）
2. 测试（运行测试套件）
3. 审查（执行跨领域的完整代码审查）

**/review** - 一次性触发涵盖所有领域的代码审查

**完成工作流**（"准备提交 PR"时自动执行）：
1. 运行测试套件
2. 从分支名中提取工单 ID
3. 由 QA 代理验证验收标准
4. 由 PM 代理生成可直接复制的 PR 描述
5. 由"图书管理员"代理捕获任何值得记忆的内容

## 核心金句

> AI 输出内容的质量，与你为其提供的上下文质量成正比。

> 结构化胜过非结构化智能。一个组织良好的专家团队，即使底层是同一模型，其产出结果也始终优于单一通用助手。

> 约束使专业化成为现实。若无约束，每个代理都会倾向于处理所有事务，最终你将回到"凭感觉编码"的老路。

> 记忆具有复利效应。一个能在会话间持久存在的简单指令文件，意味着系统每周都在变得更好。

> 引导胜于生成。当 AI 帮助笔者找到答案，而非直接给出答案时，笔者学得更多，代码也写得更好。

> 为可移植性而构建。从知识库开始，再在其上添加代理结构。即使明天需要更换工具，知识也能随之迁移。

> 不是替代，而是增强。

## 如何开始

1. **记录你的反模式** - 哪些错误是你反复犯的？将它们按严重等级记录在速查文件中
2. **添加一两个专家** - 选择你最常遇到问题的两个领域，编写代理定义
3. **添加一个简单的记忆文件** - 仅是一个用于记录指令的 Markdown 文件
4. **自动化一个工作流** - 选择你在每次提交 PR 前都会做的事情，将其脚本化
5. **逐步扩展** - 当你感受到缺口时再添加代理，而非在此之前
